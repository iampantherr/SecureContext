# SecureContext Agent Harness — Token-Optimized Workflow Rules

**Version:** 0.10.0+
**Applies to:** every agent using the `zc-ctx` MCP plugin (Claude, GPT, Gemini, and any other LLM).

These are the rules an agent should follow to maximize token efficiency while working with SecureContext. They convert SC from a *tool that's available* into a *tool that's the default*. Measured savings: **~80% reduction in context overhead** on typical multi-session project work.

Copy this file's *Hard Rules* section into your global CLAUDE.md (or equivalent for your agent system). The installer (`node install.mjs`) can do this for you.

---

## The two-tier knowledge model

**Tier 1 — Knowledge layer.** Lives inside SC. Contains:
- Per-file L0 (purpose) + L1 (detail) summaries generated by a local coder LLM
- Per-project orientation card (`zc_project_card`)
- Decision ledger (`zc_remember`)
- Tool-output archive (bash/test/build output auto-captured into KB)
- Session summaries (`zc_summarize_session`)

**Tier 2 — Raw files.** Live on disk. Expensive to read; always current.

**The rule:** *Tier 1 is the default. Tier 2 is the exception, and only to write.*

Any question that starts with *check*, *verify*, *review*, *remind me*, *what does X do*, *is X still in the code* → **Tier 1**. Any action that ends in *Edit*, *Write*, or *Patch* → **Tier 2**.

---

## Session lifecycle — the 4-step opening

Every session, in order:

1. **`zc_recall_context()`** — restore working memory (up to 100 facts), recent events, status.
2. **`zc_project_card()`** — get the 500-token orientation card (stack, layout, state, gotchas, hot files).
3. **`zc_search([relevant_queries])`** — find prior work on the current topic before exploring.
4. *(First-time projects only)* **`zc_index_project()`** — bulk-index every source file with semantic L0/L1 summaries. Run once per project; the PostEdit hook keeps it fresh after that.

Do not `ls`, `Glob`, or `Read CLAUDE.md` to orient. Steps 1–2 replace all of that at ~1/20th the token cost.

---

## During the session — the default verbs

| If you need to… | Use this, not Read |
|---|---|
| Understand what a file does | `zc_file_summary(path)` |
| Find where X is implemented | `zc_search(["X implementation"])` |
| Check if a pattern exists anywhere | `zc_check(question)` |
| See what changed recently | `git diff HEAD~1 -- path` (small) |
| Restore prior decisions | `zc_recall_context()` |
| See what a past command produced | `zc_search(["command name"])` (its output was auto-captured) |
| Verify a test passed | Check the PostBash captured summary (auto-stored) |

`Read` is for one thing: **you are about to `Edit` or `Write` that exact file.** Read only the byte range you will touch (use `offset` / `limit`).

---

## During the session — what to persist

| When | Call |
|---|---|
| Architectural decision | `zc_remember(key, value, importance=5)` |
| Non-obvious gotcha discovered | `zc_remember(key, value, importance=5)` |
| Commit hash of a checkpoint | `zc_remember("ckpt_${task}", hash, 4)` |
| Test result / build pass | `zc_remember("test_pass_${feature}", details, 3)` |
| State of the project changed meaningfully | `zc_project_card({state: "…"})` |
| Read a web doc | `zc_fetch(url)` — content goes to KB, not context |

Don't wait until end of session. Facts lost to context-window eviction are irrecoverable.

---

## End-of-session

1. **`zc_summarize_session()`** — persist work to archival memory. Next session restores from it.

## Hard rules (violations waste tokens; many are hook-enforced)

1. **Never Read the same file twice in one session** unless you wrote to it. The PreRead hook blocks duplicates and redirects you to `zc_file_summary`.
2. **Never re-run a bash command to "check status."** The PostBash hook archives every large output to KB; use `zc_search` to retrieve it.
3. **Never paste whole files into your response.** Reference the L0/L1 summary or a specific line range.
4. **Never decline a request with "I don't have context."** Before declining, run `zc_recall_context()` and `zc_search` for relevant topics. The answer may already be in memory.
5. **Never skip session startup** (steps 1–3). The cost of running them is ~2k tokens; the cost of skipping is typically ~20–50k in re-orientation.
6. **Never index via first-N-char truncation when the semantic summarizer is available.** The PreEdit hook / `zc_index_project` auto-regenerate via Ollama. Truncation is a silent cost: agents realize the summary isn't informative and Read the full file anyway — *worse* than no summary.

## Semantic summaries — local Ollama

SC v0.10.0+ generates L0/L1 summaries using a local Ollama coder model. No cloud calls, no data leaves your machine.

### Recommended model

```
ollama pull qwen2.5-coder:14b     # sweet spot: ~8GB VRAM, ~5-8s/file, excellent code understanding
```

### Alternatives (trade-offs)

| Model | VRAM | Speed | Quality |
|---|---|---|---|
| `qwen2.5-coder:14b` | 8 GB | fast | **best default** |
| `qwen2.5-coder:7b` | 4 GB | faster | very good |
| `qwen2.5-coder:32b` | 20 GB | slow | best quality (overnight indexing) |
| `qwen2.5:3b` | 2 GB | fast | decent fallback |

Auto-probe selects the first available from the preferred list. Override with `ZC_SUMMARY_MODEL=...` in your MCP env block.

### VRAM lifecycle

Default `ZC_SUMMARY_KEEP_ALIVE=30s` — Ollama unloads the model from VRAM 30 seconds after the last request. During a burst (indexing, PostEdit refresh) it stays hot. Idle = no VRAM occupation.

### Fallback

If Ollama is unreachable or no suitable model is installed, the summarizer falls back to deterministic truncation (first 100 / 1500 chars). Indexing never blocks on LLM failure. You'll see a one-line warning pointing at the fix.

## For operators: install.mjs wiring

`install.mjs` offers to:

1. Register the MCP server in `~/.claude.json`
2. Install hooks into `~/.claude/hooks/` and register them in `~/.claude/settings.json`
3. Append harness rules (this section) to `~/.claude/CLAUDE.md`

All three are opt-in. Uninstall cleanly via `node install.mjs --uninstall`.

## Token-savings evidence

Measured on a typical 10-session project:

| Scenario | Baseline (no SC) | SC v0.9.0 + discipline | SC v0.10.0 harness |
|---|---|---|---|
| Session startup | ~8k tok | ~5k | **~2k** |
| "What does X do?" | ~4k | ~2k | **~0.4k** |
| Bug-fix session (5 files) | ~45k | ~24k | **~8k** |
| Heavy bash session | ~160k | ~160k | **~1k** |
| 10-session project total | ~510k | ~200k | **~100k** |

Savings come from three sources: (a) semantic summaries replace raw file reads, (b) auto-captured bash output replaces re-running commands, (c) the project card replaces orientation rituals.
